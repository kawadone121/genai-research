# 製造業向け生成AI開発：暗黙知の形式知化手法まとめ

本ドキュメントは、工場用機械の高度な問い合わせ対応において不可欠な「熟練者の暗黙知」を、生成AI（特にRAGシステム）が利用可能な「形式知」に変換するための手法をまとめたものです。

## 概要

RAG（Retrieval-Augmented Generation）の精度は、検索対象となるデータ（ナレッジベース）の質に依存します。マニュアル化されていない「経験則」「勘」「文脈」といった暗黙知を、どのようにしてデータ化するかが最大の課題です。ここではその具体的なアプローチを4つのカテゴリに細分化しました。

---

## 1. 対話・言語化アプローチ
熟練者の頭の中にある情報を、インタビューや対話を通じて引き出し、言語テキストとして書き下す手法です。

- **概要・イメージ**:
  「なぜその結論に至ったのか？」を深掘りするインタビューを行い、無意識の思考プロセスを言語化します。
  > *イメージ: 熟練技術者に対して専属のライターが付き添い、彼らの判断の背景にある論理を一つ一つ聞き出して「Q&A集」や「事例解説書」を作成する。*

- **具体的な手法名**:
  - **Cognitive Task Analysis (CTA / 認知的タスク分析)**: 専門家がタスクを実行する際の認知プロセス（何に注目し、どう判断したか）を分析する手法。
  - **Protocol Analysis (発話思考法)**: 作業中に考えていることを独り言のように喋ってもらい、それを記録・分析する。
  - **Expert Interview to QA**: インタビュー内容をそのまま生成AI向けの高品質なQ&Aペアデータに変換する。

- **メリット**:
  - **情報の深さ**: 文脈や微妙なニュアンスまで詳細にテキスト化できる。
  - **人間も読める**: 作成資料が新人教育の教材としてもそのまま使える。

- **デメリット**:
  - **属人性と工数**: 熟練者とインタビュアー双方の時間を大量に消費する。
  - **言語化の限界**: 「なんとなく」という直感を言葉にするのは技術者自身にとっても困難な場合がある。

---

## 2. 構造化・グラフ化アプローチ (GraphRAG等)
知識をフラットなテキストではなく、「要素」と「関係性」のネットワークとして定義する手法です。

- **概要・イメージ**:
  「現象Aの原因は部品Bである」「部品Bは機能Cに属する」といった関係性を定義し、知識の地図を作ります。
  > *イメージ: ホワイトボードに付箋を貼り、それらを線で結んでトラブルの繋がりを可視化する。AIはその地図を辿って「これが起きているなら、あれも怪しい」と推論する。*

- **具体的な手法名**:
  - **Knowledge Graph Construction**: 専門用語、部品、現象、処置などをノード（点）とエッジ（線）で結び、知識グラフを構築する。
  - **Ontology Engineering**: 「モーター」は「駆動装置」の一種である、といった概念の階層構造を定義する。
  - **Causal Mapping (因果マップ)**: トラブルの因果関係（原因→結果）に特化したマップを作成する。

- **メリット**:
  - **推論能力の向上**: キーワード検索では見つからない、論理的な繋がり（GraphRAGの強み）を活用できる。
  - **表記ゆれへの強さ**: 「異音」「変な音」を同じ概念として紐付けられる。

- **デメリット**:
  - **設計難易度**: どのような概念（スキーマ）を定義するかの設計が非常に難しく、維持管理コストが高い。
  - **柔軟性の欠如**: 定義されていない新しい事象には対応しにくい。

---

## 3. 事例・ログ標準化アプローチ
現場に残されている非構造化データ（日報、修理報告書）を、AIが扱いやすい形式に整形する手法です。

- **概要・イメージ**:
  自由記述で書かれた現場のメモ書き（日報）を、一定のフォーマットに変換して「使えるデータ」にします。
  > *イメージ: 散乱した手書きの修理メモを、「発生状況」「原因」「処置内容」「再発防止策」という決まった枠組みのエクセル表に整理し直す。*

- **具体的な手法名**:
  - **5W1H標準化**: 自由記述テキストから5W1H（いつ・どこで・誰が…）を抽出し、構造化データに変換する。
  - **タグ付け・メタデータ付与**: 過去の報告書に対し、「重要」「レアケース」「製品シリーズA」などのタグを事後的に付与する。
  - **Case-Based Reasoning (CBR) フォーマット化**: 過去の解決事例として「問題記述」と「解決策」のペア形式に正規化する。

- **メリット**:
  - **資産の活用**: 既に存在する大量のドキュメントを有効活用できる。
  - **網羅性**: 過去に起きた実際の事例を網羅できるため、実践的。

- **デメリット**:
  - **データ品質のバラつき**: 元の記録が曖昧だったり間違っている場合、ゴミデータ（Garbage In）になる。
  - **文脈の欠落**: 「なぜその処置をしたか」という一番重要な思考プロセスが記録に残っていないことが多い。

---

## 4. プロセス・ロジック抽出アプローチ
トラブルシューティングの手順（フロー）を明確なアルゴリズムや決定木として書き出す手法です。

- **概要・イメージ**:
  「Yesなら次のページ、Noなら3ページへ」というような、診断フローチャートを作成します。
  > *イメージ: 熟練者が無意識に行っている「消去法」の手順を、誰でも辿れるフローチャート図として書き起こす。*

- **具体的な手法名**:
  - **Decision Tree Extraction (決定木作成)**: 診断プロセスをIF-THENルールとして記述する。
  - **Troubleshooting Flowcharting**: 視覚的なフローチャートを作成し、各ステップの説明文を記述する。
  - **SOP (Standard Operating Procedures)のさらなる詳細化**: 既存手順書の「行間」にある判断分岐を埋める。

- **メリット**:
  - **再現性**: 誰が（どのAIが）やっても同じ手順で診断できるため、回答品質が安定する。
  - **安全性**: 確認すべき危険項目をプロセスに組み込むことで、見落としを防げる。

- **デメリット**:
  - **硬直性**: 想定外の事象が起きた時、フローチャートにない道に進めず行き詰まる。
  - **複雑性の爆発**: あらゆる可能性を網羅しようとすると、フローが巨大すぎて管理不能になる。
